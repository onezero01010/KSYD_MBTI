# KSYD 2026 프로그램3 — "카나의 일꾼 유형 테스트" 웹앱 개발 계획서

> 이 문서는 **AI 코딩 에이전트가 그대로 실행**할 수 있도록 작성된 명세서입니다.
> 모든 콘텐츠(문항·선택지·결과)와 채점 로직, 디자인 가이드, 단계별 작업 목록, 완료 기준이 포함되어 있습니다.

---

## 0. 프로젝트 개요

- **행사**: 살레시오 수도회 청년 대회 **KSYD 2026**
- **대회 주제**: 카나의 혼인잔치 — "그가 시키는 대로 하여라: 신앙인, 봉사하는 자유인"
- **프로그램3 목적**: "봉사하는 자유인"으로서, 각자가 살레시오 안에서 하고 있는 활동을 돌아보고 **자기 자신을 성찰**하도록 돕는다.
- **방식**: MBTI를 레퍼런스로, 카나의 혼인잔치 상황 속 12개의 이지선다 질문에 답하면 **16가지 '일꾼 유형'** 중 하나로 안내하고, 그 유형을 **나의 실제 봉사 활동에 대한 성찰**로 연결한다.
- **레퍼런스 UX**: https://kongbiti.netlify.app/ (모바일 우선, 한 문항씩 진행, "분석중.." 로딩, 저장 가능한 결과 카드, 다시하기 / 모든 유형 보기 / 공유)
- **동시 사용자**: 약 50명 동시 접속 (행사 현장에서 각자 휴대폰으로 동시에 진행)

## 1. 핵심 요구사항 (Must-have)

1. 시작 페이지 1개 → 질문 페이지 12개 → 로딩(분석중) → 결과 페이지 1개 (16유형 중 1개 표시).
2. 각 질문은 **이지선다(A/B)**. 한 화면에 1문항.
3. 12문항을 채점해 16개 MBTI형 일꾼 유형 중 하나를 산출.
4. 결과 페이지에 유형명·설명·**성찰 질문**·살레시오 봉사 연결 메시지 표시.
5. 결과를 **이미지로 저장/스크린샷**하기 좋게 카드 형태로 구성.
6. **모바일 우선** 반응형. 50명 동시 접속에도 안정적(정적 호스팅).
7. 진행률 표시(예: 3/12), 이전/다음 이동.
8. "다시하기", "모든 유형 보기" 기능.

### 선택 요구사항 (Nice-to-have, 별도 플래그로 on/off)

- 결과 공유 링크(URL에 결과 코드 인코딩).
- **진행자용 집계 대시보드**(현장에서 "우리 그룹엔 어떤 일꾼이 많나" 함께 나누기용) — *백엔드 필요, §10 참고*.
- 입장용 QR 코드 이미지 생성.
- 간단한 효과음/애니메이션(켜고 끌 수 있게).

## 2. 기술 스택 및 아키텍처

### 권장 스택
- **Vite + React + TypeScript**
- **Tailwind CSS** (빠른 반응형/디자인 토큰)
- **React Router** (페이지 라우팅: `/`, `/q/1`~`/q/12`, `/result/:typeCode`, `/types`)
- 상태 관리: React 내장 `useState`/`useReducer` + `Context` (외부 라이브러리 불필요)
- 결과 카드 이미지 저장: `html-to-image` 또는 `dom-to-image`로 PNG 다운로드 버튼 제공 (모바일에서는 "이미지 꾹 눌러 저장" 안내도 병행)
- 배포: **Netlify** (정적 빌드, CDN 자동)

> 더 단순하게 가고 싶다면 **순수 HTML/CSS/JavaScript 단일 페이지**로도 가능. 라이브러리 의존성이 줄어 현장 안정성이 높음. 에이전트는 위 React 스택을 1순위로 하되, 요청 시 vanilla 버전으로 대체할 수 있도록 컴포넌트 경계를 명확히 둘 것.

### 왜 이 구조로 50명 동시접속이 가능한가 (에이전트가 이해할 것)
- 퀴즈 진행·채점이 **전부 브라우저에서** 실행됨 → 서버 연산/DB 조회 없음.
- 배포물은 정적 파일(HTML/CSS/JS/이미지)뿐 → Netlify CDN이 각 사용자에게 캐싱 전송.
- 따라서 동시 사용자 수는 사실상 호스팅 CDN 한도에 좌우되며, **50명은 여유롭게 처리**됨.
- 집계 대시보드(선택)를 켤 때만 가벼운 백엔드가 필요하며, 그 경우에도 무료 티어로 50명은 충분.

### 데이터 분리 원칙
- 문항/결과 콘텐츠는 **코드와 분리**된 데이터 파일(`questions.ts`, `results.ts` 또는 JSON)로 관리.
- 운영자가 문구를 쉽게 수정할 수 있도록 콘텐츠는 한곳에 모은다.

## 3. 전체 사용자 흐름 (User Flow)

```
[시작 페이지]
   └─(시작하기 버튼)→ [Q1] → [Q2] → ... → [Q12]
                                              └─(마지막 선택)→ [분석중.. 로딩 ~1.5초]
                                                                      └─→ [결과 페이지: 16유형 중 1개]
                                                                              ├─ 다시하기 → [시작 페이지]
                                                                              ├─ 모든 유형 보기 → [유형 목록]
                                                                              └─ 결과 저장 / 공유
```

- 각 질문 화면: 상단 진행률, 상황 문구, (A)/(B) 두 개의 큰 버튼, 이전으로 가기.
- 선택 즉시 다음 문항으로 부드럽게 전환(슬라이드/페이드).
- 뒤로 가면 직전 선택이 유지되도록 답안을 상태에 저장.

## 4. 페이지 구조 및 라우팅

| 경로 | 화면 | 설명 |
|---|---|---|
| `/` | 시작 페이지 | 타이틀, 대회 주제 한 줄, "시작하기" 버튼 |
| `/q/:n` (1~12) | 질문 페이지 | 문항 n 표시, A/B 선택, 진행률 |
| `/loading` | 분석중 | 1~2초 로딩 애니메이션 후 결과로 이동 |
| `/result/:typeCode` | 결과 페이지 | 산출된 유형 카드 + 성찰 + 버튼들 |
| `/types` | 모든 유형 보기 | 16유형 카드 그리드 |

- `typeCode`는 4자리 MBTI 코드(예: `ENFJ`). URL로 결과 공유 가능.
- 답안 미완료 상태로 `/result`에 직접 접근하면 `/`로 리다이렉트.

## 5. 채점 알고리즘 (정확히 구현할 것)

### 5.1 지표 배정 (각 지표 3문항씩, 총 12문항)

| 지표 | 해당 문항 번호 |
|---|---|
| **S / N** | Q1, Q6, Q7 |
| **T / F** | Q2, Q4, Q11 |
| **J / P** | Q3, Q8, Q12 |
| **E / I** | Q5, Q9, Q10 |

### 5.2 선택지 → 글자 매핑 규칙 (전 문항 공통)
- **모든 문항에서 (A) = 지표의 앞 글자, (B) = 지표의 뒤 글자.**
  - 예: `[S/N]` 문항 → (A)=S, (B)=N
  - 예: `[T/F]` 문항 → (A)=T, (B)=F
  - 예: `[J/P]` 문항 → (A)=J, (B)=P
  - 예: `[E/I]` 문항 → (A)=E, (B)=I
- 각 문항 데이터에는 `dimension`("SN"|"TF"|"JP"|"EI")과 `optionA.pole`, `optionB.pole`을 명시(부록 데이터 참고).

### 5.3 산출 로직 (의사코드)
```text
counts = { S:0, N:0, T:0, F:0, J:0, P:0, E:0, I:0 }

for each answered question q:
    pole = (선택이 A) ? q.optionA.pole : q.optionB.pole   // 예: "S" 또는 "N"
    counts[pole] += 1

axis1 = counts.E >= counts.I ? "E" : "I"   // E/I
axis2 = counts.S >= counts.N ? "S" : "N"   // S/N
axis3 = counts.T >= counts.F ? "T" : "F"   // T/F
axis4 = counts.J >= counts.P ? "J" : "P"   // J/P

typeCode = axis2 + ... // 주의: MBTI 표기 순서는 E/I, S/N, T/F, J/P
typeCode = axis1 + axis2 + axis3 + axis4   // 예: "E"+"N"+"F"+"J" = "ENFJ"
```
- 지표마다 문항이 3개(홀수)이므로 **동점은 발생하지 않음** → 16유형 모두 도달 가능.
- (안전장치) 혹시라도 동점이 생기면 위 비교에서 앞 글자 우선(`>=`)으로 결정.
- 최종 `typeCode`(예: `ENFJ`)로 결과 데이터(부록 B)를 조회해 표시.

## 6. UI/UX 디자인 가이드

### 톤 & 무드
- **밝고 따뜻하며 잔치 같은** 분위기. 가톨릭/살레시오 청년 행사에 어울리는 정갈하면서도 친근한 느낌.
- 카나의 혼인잔치 모티프: **포도/포도주, 물독, 잔치 테이블, 따뜻한 빛**.
- 권장 팔레트(예시, 자유 조정 가능):
  - 메인: 와인/버건디 계열 `#7B2D3B`
  - 보조: 골드/앰버 `#E0A458`, 크림 `#FBF3E4`
  - 텍스트: 짙은 갈색 `#3A2A24`
- 폰트: 가독성 좋은 한글 산세리프(예: Pretendard, Noto Sans KR). 제목은 약간 라운드/친근한 느낌.

### 레퍼런스에서 가져올 UX 패턴 (kongbiti)
- 모바일 우선, 단일 컬럼, 큰 터치 버튼.
- 한 문항씩 진행, 선택 즉시 전환.
- 마지막에 "분석중.." 로딩 한 박자.
- 결과는 **저장/스크린샷 친화적인 카드** + "이미지 꾹 눌러 저장하기" 안내.
- 결과 하단: "다시하기", "모든 유형 보기", "공유".

### 화면별 핵심 요소
- **시작 페이지**: 제목("카나의 혼인잔치, 나는 어떤 일꾼?") + 부제(대회 주제 한 줄) + 일러스트/이모지 + 큰 "시작하기" 버튼. 소요 시간 안내("약 2분").
- **질문 페이지**: 상단 진행률 바(n/12)와 Phase 표시(직면→실천→변화), 상황 문구, A/B 두 개의 큰 카드형 버튼, "이전" 텍스트 버튼.
- **결과 페이지**: §8 참고.

### 접근성
- 충분한 명도 대비, 버튼 최소 44px 터치 영역, 글자 확대에도 깨지지 않는 레이아웃.
- 색만으로 정보 전달 금지(아이콘/텍스트 병행).

## 7. 콘텐츠 구조: 3개의 Phase

질문 순서는 **카나의 혼인잔치 서사**를 따른다(이 순서를 그대로 페이지 순서로 사용).

- **Phase 1. 예상치 못한 위기와 성모님의 알아챔 (직면의 여정)** — Q1~Q6
- **Phase 2. "그가 시키는 대로 하여라" (순명과 실천의 여정)** — Q7~Q9
- **Phase 3. 기적의 발견과 기쁜 파견 (변화의 여정)** — Q10~Q12

질문 페이지 상단에 현재 Phase 이름을 작게 표기하면 서사 몰입에 도움이 된다(선택).

## 8. 결과 페이지 상세 명세 (프로그램 목적 반영)

결과 페이지는 단순 "MBTI 결과"를 넘어, **프로그램3의 성찰 목적**을 담아야 한다.

각 결과 카드 구성:
1. **유형 코드 + 일꾼 유형명** (예: `ENFJ` · 기쁜파견꾼)
2. **한 줄 설명** (부록 B)
3. **이모지/심볼** (유형 분위기를 나타내는 1~2개 이모지로 가볍게 — 별도 일러스트 제작 전 임시)
4. **성찰 연결 문단** (아래 템플릿으로 생성): 카나의 일꾼으로서의 모습을 → **실제 살레시오 봉사 활동에 대한 성찰**로 잇는다.
5. **성찰 질문 2~3개** (그룹 나눔/개인 묵상용)
6. 버튼: 결과 저장(PNG) · 다시하기 · 모든 유형 보기 · 공유

### 성찰 문단 템플릿 (에이전트가 16유형 각각에 맞춰 1~2문장으로 채울 것)
> "{유형명}인 당신은 카나의 잔치에서 {유형의 강점 행동}을 통해 봉사합니다.
> 지금 내가 살레시오 공동체 안에서 하고 있는 활동을 떠올려 보세요.
> 나는 어디에서 이런 모습으로 '봉사하는 자유인'이 되고 있나요?"

### 공통 성찰 질문 (모든 결과에 노출, 그룹 나눔용)
- 나는 카나의 잔치에서 어떤 자리에 서 있었나요? 지금 내 봉사의 자리와 닮았나요?
- "그가 시키는 대로 하여라"는 말씀을, 내 활동 안에서 어떻게 살고 있나요?
- 내 유형의 강점이 공동체에 기쁨이 된 순간이 있었나요?

> 참고: 16유형별 **맞춤 성찰 질문 1개씩**을 부록 B에 추가해도 좋다(에이전트가 유형 설명을 근거로 자연스럽게 생성).

## 9. 데이터 모델 (구현용 스키마)

### 9.1 질문 스키마
```ts
type Pole = "S" | "N" | "T" | "F" | "J" | "P" | "E" | "I";

interface Option {
  label: "A" | "B";
  text: string;
  pole: Pole;
}

interface Question {
  id: number;            // 1~12 (페이지 순서)
  phase: 1 | 2 | 3;
  phaseTitle: string;
  dimension: "SN" | "TF" | "JP" | "EI";
  prompt: string;        // 상황 문구
  optionA: Option;       // pole = 지표 앞 글자
  optionB: Option;       // pole = 지표 뒤 글자
}
```

### 9.2 결과 스키마
```ts
interface ResultType {
  code: string;          // "ENFJ" 등 16개
  name: string;          // "기쁜파견꾼" 등
  summary: string;       // 한 줄 설명
  emoji: string;         // 임시 심볼
  reflection: string;    // 성찰 연결 문단 (§8 템플릿 기반)
  reflectionQuestion: string; // 유형 맞춤 성찰 질문 1개
}
```

## 10. (선택) 진행자용 집계 대시보드

> 현장에서 "우리 그룹엔 어떤 일꾼이 많은지" 함께 나누고 싶을 때만 구현. **기본은 OFF**, 환경변수 플래그로 켠다.

- 가장 간단한 구현: **Supabase**(무료) 또는 **Firebase Firestore**(무료) 테이블 1개.
  - 결과 페이지 도달 시 `{ typeCode, timestamp, (선택)groupName }` 1행 insert.
  - 진행자 화면 `/dashboard`에서 유형별 카운트 실시간 집계(막대 그래프).
- 개인정보는 수집하지 않음(이름/연락처 X). 유형과 시각, 선택적 그룹명만.
- 50명 규모는 무료 티어로 충분.
- 대안(백엔드 없이): 결과 페이지에서 유형 코드를 보여주고, 참가자가 진행자에게 구두/거수로 알려 진행자가 수동 집계.

- 첫 로딩 가벼움 유지(이미지 최적화, 폰트 서브셋).
- 오프라인/약한 와이파이 대비: 핵심 자산 캐싱(가능하면 PWA/서비스워커는 선택).
- 새로고침해도 진행 상태가 너무 쉽게 날아가지 않게(주의: `localStorage`/`sessionStorage`는 **이 명세에서 허용** — 단, 사용자가 직접 만든 배포본 한정. 결과 공유는 URL의 `typeCode`로 처리).
- 모든 인터랙션 키보드 접근 가능.
- 결과 PNG 저장은 모바일 사파리/크롬에서 테스트.

## 12. 배포

1. `npm run build`로 정적 빌드.
2. **Netlify**에 연결(Git 푸시 시 자동 배포) 또는 빌드 폴더 드래그 업로드.
3. SPA 라우팅 폴백 설정: `public/_redirects`에 `/* /index.html 200`.
4. 커스텀 도메인/서브경로가 필요하면 `vite.config`의 `base` 설정 확인.
5. OG 태그(제목·설명·썸네일) 설정해 카톡/인스타 공유 시 미리보기 노출(레퍼런스처럼).

## 13. AI 에이전트용 단계별 작업 분해 (Task Breakdown)

> 각 단계는 독립적으로 검증 가능하도록 작은 커밋 단위로 진행할 것.

1. **프로젝트 셋업**: Vite+React+TS+Tailwind 초기화, 라우터 설치, 폴더 구조 생성.
2. **콘텐츠 데이터 입력**: 부록 A(질문 12개), 부록 B(결과 16개)를 `data/`에 정확히 타이핑. *문구 오탈자 없이 그대로.*
3. **공통 레이아웃/디자인 토큰**: 팔레트·폰트·버튼·카드 컴포넌트.
4. **시작 페이지** 구현.
5. **질문 페이지 컴포넌트**(1개) + 12문항 데이터로 동작, 진행률·이전/다음·Phase 표시.
6. **상태 관리**: 답안 저장(Context), 뒤로가기 시 유지.
7. **채점 함수** 구현(§5 의사코드) + 단위 테스트(아래 §14의 케이스 포함).
8. **분석중 로딩** 화면(1~2초) → 결과로 이동.
9. **결과 페이지**: 유형 카드 + 성찰 문단/질문 + 버튼(저장/다시/공유/모든유형).
10. **모든 유형 보기**(`/types`) 그리드.
11. **결과 PNG 저장** 기능 + 모바일 안내.
12. **반응형·접근성·OG 태그** 마감.
13. (선택) 집계 대시보드 플래그 구현.
14. **빌드 & Netlify 배포 설정**(`_redirects` 포함).
15. **QA 체크리스트**(§14) 전수 확인.

## 14. 완료 기준 (Acceptance Criteria) & QA 체크리스트

**기능**
- [ ] 시작 → 12문항 → 로딩 → 결과까지 끊김 없이 진행된다.
- [ ] 각 문항은 한 화면에 1개, A/B 두 선택지가 명확하다.
- [ ] 진행률(n/12)과 Phase 표기가 정확하다.
- [ ] "이전"으로 가면 직전 선택이 유지된다.
- [ ] 12문항 모두 답해야 결과가 나온다(미완료 시 결과 직접 접근 차단).

**채점 정확성 (테스트 케이스 — 반드시 통과)**
- [ ] 모든 문항 **(A)** 선택 → 각 지표 앞 글자 → `ESTJ`.
  - (S/N→S, T/F→T, J/P→J, E/I→E ⇒ E S T J)
- [ ] 모든 문항 **(B)** 선택 → 각 지표 뒤 글자 → `INFP`.
  - (S/N→N, T/F→F, J/P→P, E/I→I ⇒ I N F P)
- [ ] 임의 혼합 입력 시 §5 다수결 규칙대로 4글자가 산출된다(동점 미발생 확인).
- [ ] 산출된 `typeCode`가 부록 B의 16유형 중 정확히 하나에 매핑된다.

**표시/성찰**
- [ ] 결과에 유형명·설명·**성찰 문단·성찰 질문**이 모두 보인다.
- [ ] 결과 카드가 이미지로 저장/스크린샷하기 좋게 구성된다.
- [ ] "다시하기", "모든 유형 보기", "공유"가 동작한다.

**현장 안정성**
- [ ] 모바일(아이폰/안드로이드) 크롬·사파리에서 정상 동작.
- [ ] 50명 동시 접속 가정(정적 호스팅)에서 지연 없음.
- [ ] SPA 새로고침/딥링크 시 라우팅 폴백 동작.

## 부록 A. 질문 데이터 (12문항, 페이지 순서 = 표기 순서)

> 형식: **번호. [지표] 상황 / (A) … [pole] / (B) … [pole]**
> 규칙 재확인: 모든 문항에서 **(A)=앞 글자, (B)=뒤 글자**.

### Phase 1. 예상치 못한 위기와 성모님의 알아챔 (직면의 여정)

**Q1. [S/N] 예식 중 테이블이 조금 기울어진 것 같다. 당신은?**
- (A) 헉 테이블 약간 기울어졌는데 여기에 뭘 받치면 되겠다! → **S**
- (B) 테이블? 테이블이 기울어졌었어? 나는 몰랐네… → **N**

**Q2. [T/F] 예식 중 작은 실수가 발생한 것을 보았을 때 당신은?**
- (A) 이렇게 해야 효율적이지 않나? 대처가 좀 아쉽다. → **T**
- (B) 신랑 신부 당황하는 거 아냐? 내가 다 걱정되네. → **F**

**Q3. [J/P] 잔치 식순이 갑자기 바뀌어야 하는 상황이 온다면?**
- (A) 원래 식사부터 하는 거 아냐? 이유가 있어서 순서가 바뀌었나보네 → **J**
- (B) 오 이거부터 하는구나~ 오히려 좋아~ 재밌겠다! → **P**

**Q4. [T/F] 포도주가 바닥난 위기 상황을 알게 되었을 때 당신의 반응은?**
- (A) 헉 큰일이다! 나라도 어디 가서 포도주라도 사와야 하나? → **T**
- (B) 혼주들 준비 열심히 했을텐데… 가서 위로라도 해줘야겠다. → **F**

**Q5. [E/I] 위기 상황에서 동료 일꾼들과 소통할 때 당신은?**
- (A) "여러분 포도주가 없대요!" 즉시 상황을 알리고 대책을 논의한다. → **E**
- (B) 일이 해결될 거라고 믿고 상황을 지켜보며 지시를 기다린다. → **I**

**Q6. [S/N] "포도주가 없구나" 하시는 성모님의 말씀을 들었을 때 당신은?**
- (A) "포도주가 없다고? 그럼 이제 잔치 끝인가?" 걱정이 든다. → **S**
- (B) "성모님이 나서신 거 보니 뭔가 대단한 일이 일어날 것 같아!" → **N**

### Phase 2. "그가 시키는 대로 하여라" (순명과 실천의 여정)

**Q7. [S/N] 예수님이 "물독에 물을 채워라"고 하셨다. 나의 머릿속은?**
- (A) '흘리지 말고 정확히 채워야지!' → **S**
- (B) '이 물이 어떻게 변할까?' → **N**

**Q8. [J/P] 이제 우물가로 물을 뜨러 가야 한다. 나의 이동 스타일은?**
- (A) '최단 동선으로 가자!' → **J**
- (B) '보이는 것부터 하자!' → **P**

**Q9. [E/I] 무거운 물독을 채우는 지루한 노동 중이다. 나는 지금?**
- (A) "여러분, 조금만 더 힘내요!" → **E**
- (B) 말없이 내 몫에 집중한다. → **I**

### Phase 3. 기적의 발견과 기쁜 파견 (변화의 여정)

**Q10. [E/I] 헐 대박! 물이 포도주로 변했다! 나의 첫 행동은?**
- (A) "대박! 이것 좀 보세요!" → **E**
- (B) '와… 진짜 일어났네.' → **I**

**Q11. [T/F] 과방장이 "포도주 맛 실화냐? 최고다!"라고 칭찬한다. 이때 드는 생각은?**
- (A) '말씀대로 하니까 됐네!' → **T**
- (B) '우리를 써주시다니 감사하다.' → **F**

**Q12. [J/P] 잔치가 끝나고 드디어 뒷정리 시간! 지친 나는?**
- (A) '정해진 순서대로 빨리 치우자!' → **J**
- (B) "오늘 진짜 대박이었죠?" → **P**

## 부록 B. 결과 데이터 (16유형)

> `code` · `name` · `summary`. (성찰 문단/질문은 §8 템플릿으로 에이전트가 생성)

| code | name | summary |
|---|---|---|
| ESTP | 번쩍해결꾼 | 위기가 오면 누구보다 빠르게 움직이는 현장 해결형 일꾼 |
| ISTP | 잠잠해결꾼 | 말은 적지만 필요한 순간 조용히 문제를 고치는 실속형 일꾼 |
| ESFP | 흥잔치꾼 | 잔치의 분위기를 살리고 모두를 웃게 만드는 에너지형 일꾼 |
| ISFP | 몽글위로꾼 | 말 한마디와 따뜻한 마음으로 곁을 지켜주는 감성형 일꾼 |
| ENFP | 두근상상꾼 | 작은 변화 속에서도 기적의 가능성을 먼저 떠올리는 기대형 일꾼 |
| INFP | 묵상지킴꾼 | 놀라운 순간을 마음 깊이 담아 오래 간직하는 신비형 일꾼 |
| ENTP | 잔치발명꾼 | 예상 밖의 상황도 재밌게 받아들이고 새 길을 찾는 아이디어형 일꾼 |
| INTP | 물독탐구꾼 | 기적의 흐름과 원리를 조용히 분석해보는 탐구형 일꾼 |
| ESTJ | 척척총무꾼 | 식순과 역할을 빠르게 정리해 잔치를 안정시키는 운영형 일꾼 |
| ISTJ | 꼼꼼채움꾼 | 맡은 일을 끝까지 정확하게 채워내는 성실형 일꾼 |
| ESFJ | 다정챙김꾼 | 사람들의 표정과 마음을 살피며 모두를 챙기는 배려형 일꾼 |
| ISFJ | 순명살림꾼 | 드러나지 않아도 묵묵히 자리를 지키며 섬기는 헌신형 일꾼 |
| ENFJ | 기쁜파견꾼 | 받은 은총을 사람들과 나누고 함께 기뻐하는 나눔형 일꾼 |
| INFJ | 은총알아챔꾼 | 보이지 않는 뜻과 흐름을 먼저 알아차리는 통찰형 일꾼 |
| ENTJ | 포도작전꾼 | 위기 앞에서 대책을 세우고 사람들을 이끄는 지휘형 일꾼 |
| INTJ | 카나설계꾼 | 눈앞의 사건 너머 큰 그림과 의미를 읽어내는 전략형 일꾼 |
